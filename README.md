# Redrob Candidate Finder

Ranks ~100k candidates against a job description using embeddings + rule-based signals. Outputs the top 100 best-fit candidates.

---

## How to Run

### 1. Install dependencies
```bash
pip install -r requirements.txt
```

### 2. Add the candidate data (required — NOT in this repo)
The 100k candidate pool is **not committed** (~487 MB, too large for GitHub, and it's the hackathon's data to supply). Download `candidates.jsonl` from the hackathon source and place it at exactly:

```
data/candidates.jsonl
```

Everything else the pipeline needs is already in the repo (`data/job_description.txt`, `data/sample_candidates.json`). The precompute step reads `data/candidates.jsonl` by default, so once this file is in place the two steps below just work. (To point at a different path: `python precompute_all.py --candidates /path/to/candidates.jsonl`.)

The pipeline is **two separate steps**, on purpose: a slow precompute (network allowed) that builds everything in `artifacts/`, then a fast ranking step that runs **offline**. A judge runs Step 3 once, then can run Step 4 with networking off.

### 3. Precompute — `python precompute_all.py` (run once, slow, network allowed)
```bash
python precompute_all.py            # runs all 5 precompute scripts in order
# (optional) python precompute_all.py --candidates path/to/candidates.jsonl
```
This single command runs, in order:
```
precompute/embedded_candidates.py   # text -> vectors          (~33 min)
precompute/build_features.py        # rule-based signals       (~1-2 min)
precompute/flag_honeypots.py        # impossible-profile traps (seconds)
precompute/setup_reranker.py        # download cross-encoder   (one-time, online)
precompute/build_rerank_texts.py    # compact docs for reranker (seconds)
```
Everything is written to `artifacts/`. First run downloads the embedding model (~90MB) and the cross-encoder (~120MB) — needs internet **once, here in precompute only**.

### 4. Rank — `python rank.py` (offline, CPU-only, ≤5 min)
```bash
python rank.py --out submission.csv          # the timed, network-free step
# (optional) python rank.py --out submission.csv --artifacts artifacts
```
Done. Results are in `submission.csv` (~40s). `rank.py` sets `HF_HUB_OFFLINE=1` and loads the cross-encoder from `artifacts/reranker/` — **no network, no GPU**.

> **Why split this way:** precompute has no time limit and may use the network; the ranking step (`rank.py`) must satisfy the spec's compute constraints (≤5 min, ≤16 GB, CPU-only, no network). The models are downloaded during precompute and saved into `artifacts/` so the ranking step can load them locally and offline.

Optional — see how much the cross-encoder changed the ranking vs the rule/embedding score alone:
```bash
python compare_rankings.py
```

## Project Structure

```
├── data/
│   ├── candidates.jsonl        ← NOT in repo (487MB), add manually
│   ├── job_description.txt
│   └── sample_candidates.json
├── precompute/
│   ├── embedded_candidates.py  ← Step 1: text → vectors (bi-encoder, MiniLM)
│   ├── build_features.py       ← Step 2: extract signals per candidate
│   ├── flag_honeypots.py       ← Step 3: detect fake/planted resumes
│   ├── setup_reranker.py       ← Step 4: download cross-encoder → artifacts/reranker/
│   └── build_rerank_texts.py   ← Step 5: compact per-candidate docs for reranking
├── artifacts/                  ← auto-generated by precompute scripts (incl. reranker/)
├── precompute_all.py           ← runs all precompute steps in order (one command)
├── rank.py                     ← ranking step (offline): score → shortlist → cross-encoder rerank → top 100
├── compare_rankings.py         ← optional: Phase-1 vs reranked diff
├── submission.csv              ← final output
└── requirements.txt
```

## How It Works (quick)

1. **Embed** — converts the JD and every candidate's profile into vectors using `all-MiniLM-L6-v2` (bi-encoder, for fast recall over 100k)
2. **Features** — rule-based signals: experience fit, location, notice, engagement, and **evidence-weighted** skill matching (verified hits vs keyword stuffing)
3. **Honeypots** — flags candidates with impossible dates/math (planted traps)
4. **Rank** — a **3-part score**: additive *relevance core* (cosine + verified skills + experience + product-company) × *bounded soft modifiers* (location / notice / engagement) × *hard gates* (zero out honeypots & irrelevant titles, penalize genuine mismatches)
5. **Rerank** — the top-400 shortlist is reordered by a local **cross-encoder** (`ms-marco-MiniLM-L-12-v2`) that reads the JD and candidate jointly, then blended 50/50 with the Phase-1 score → top 100. Runs CPU-only, offline, in ~40s.
